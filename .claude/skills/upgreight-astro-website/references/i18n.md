# Internationalization (i18n)

Read this **only if the project has multiple languages.** A single-language site gets `<html lang>` set once and nothing else here.

The whole risk with i18n is **lying to search engines**: hreflang tags pointing at pages that don't exist, canonicals pointing at the wrong language, or a language switcher that guesses sibling URLs which 404. Wrong i18n signals are worse than none - they can de-index pages or split ranking signals. So the rule throughout: **only ever reference translations that actually exist.** Never construct a localized URL by pattern and assume it resolves.

## Decide the URL strategy

Pick one, up front:

| Strategy | Looks like | When |
|---|---|---|
| **Path prefix, default not prefixed** | `/about/` (de) · `/en/about/` (en) | Default language is the primary market; cleanest for DACH-first sites |
| **Path prefix, all prefixed** | `/de/about/` · `/en/about/` | All languages are equal; no implicit default |
| Subdomain / ccTLD | `en.site.com` / `site.co.uk` | Rare for these projects; usually overkill |

Default to **path prefix**. Decide whether the primary language is prefixed (`prefixDefaultLocale`). Set the default locale to the client's primary market (often `de`).

## Astro i18n config

```js
// astro.config.mjs
export default defineConfig({
  site: 'https://clientdomain.com',
  i18n: {
    defaultLocale: 'de',
    locales: ['de', 'en'],
    routing: {
      prefixDefaultLocale: false,   // /about (de), /en/about (en)
    },
  },
  integrations: [sitemap({ i18n: { defaultLocale: 'de', locales: { de: 'de', en: 'en' } } })],
});
```

Giving `@astrojs/sitemap` the `i18n` option makes it emit the correct `xhtml:link` hreflang annotations in the sitemap automatically - but only for pages it can pair, so your page structure still has to actually contain both language versions.

Organize pages by locale, e.g. `src/pages/[lang]/...` driven by content, or parallel folders. Whatever you choose, a page exists in a language **only if you actually created/authored it** in that language.

## Per-page lang + localized canonical

Two rules that pages must follow:

1. **`<html lang>` matches the page's language** - `lang="en"` on English pages, set via `BaseLayout`'s `lang` prop. Don't leave every page on the default.
2. **The canonical points at the page's own localized URL** - the English about page canonicalizes to `https://site.com/en/about/`, *not* to the German one. Each language version is its own canonical; they are not duplicates of each other, they are alternates. (The `SEO` component already self-canonicalizes from `Astro.url`; just make sure each localized route renders at its real localized path.)

## hreflang - only real translations

hreflang tells Google "here's the same page in another language." Every `hreflang` URL you emit **must be a page that exists and returns 200.** If the About page has DE + EN but the Blog only has DE, then:

- About emits hreflang for `de` and `en` (both exist).
- A DE-only blog post emits **no** alternate hreflang (or only its own) - do **not** emit an `/en/...` hreflang for a translation that was never written.

Best practice when you do emit them:

- Include a **self-referencing** hreflang (the page lists itself).
- Add **`x-default`** pointing at the default-language version (or a language selector).
- Make the set **reciprocal** - if A lists B as `en`, B must list A as `de`. One-directional hreflang is ignored.

Build the `alternates` array passed to `SEO`/`BaseLayout` from your **translation model** (next section), never from string-munging the current path. If a translation doesn't exist in the model, it doesn't go in the array.

```astro
---
// Example: derive alternates from a known translation record, not by guessing
const alternates = getTranslations(Astro.url.pathname); // returns only existing ones
---
<BaseLayout {...props} lang={lang} alternates={alternates} />
```

## The language switcher

The switcher must link to the **actual** translated URL of the current page - resolved from real translation relationships, not assembled by swapping the locale segment.

Why this matters: pages don't always map 1:1. `/en/about/` might be `/ueber-uns/` in German (translated slug), `/en/blog/post-x/` might have no German version at all. A naive switcher that turns `/en/about/` into `/de/about/` links to a 404.

Correct behaviour:
- For the current page, look up its sibling translations in the translation model.
- Render a link only for languages that **have** a translation; for languages that don't, either hide that option or link to the localized homepage (and make that fallback explicit, not silent).
- Mark the current language as active/non-clickable.

```astro
---
// LanguageSwitcher.astro
const current = Astro.currentLocale;            // Astro knows the active locale
const siblings = getTranslations(Astro.url.pathname); // [{ lang, url }] - only real ones
---
<nav aria-label="Language">
  {siblings.map((s) => (
    <a href={s.url} hreflang={s.lang} aria-current={s.lang === current ? 'true' : undefined}>
      {s.lang.toUpperCase()}
    </a>
  ))}
</nav>
```

The switcher and the hreflang tags should be driven by the **same** `getTranslations` source of truth - then they can never disagree.

## Modelling translations

You need a reliable record of "which pages are translations of each other." Options, simplest first:

- **Content collections:** give each entry a shared `translationKey` (or `translationOf`) field in frontmatter. Entries with the same key are translations; query the collection by key to get the sibling URLs. This is the cleanest for blog/article-type content.
- **Static pages:** keep a small `src/i18n/routes.ts` map: `{ home: { de: '/', en: '/en/' }, about: { de: '/ueber-uns/', en: '/en/about/' } }`. `getTranslations` looks the current path up in it. Explicit and impossible to mis-guess.
- **Sanity:** use document-level translation references (a `language` field + a translation metadata document, e.g. `@sanity/document-internationalization`). Query the linked translations; emit alternates only for the languages present on that specific document.

Whichever you use, both the hreflang `alternates` and the language switcher read from it. There is **no** code path that builds a localized URL by assumption.

## Common mistakes

- ❌ Switcher swaps `/en/` → `/de/` and hopes the page exists → 404s, broken UX, broken hreflang.
- ❌ Emitting hreflang for every locale on every page regardless of whether the translation exists.
- ❌ English page canonicalizing to the German URL (tells Google EN is a duplicate of DE → EN drops out).
- ❌ Same `<html lang>` on every page.
- ❌ Non-reciprocal or missing self-referencing hreflang.
- ❌ Forgetting `x-default`.
- ✅ One translation model; hreflang + switcher + canonical all derived from it; only existing translations referenced.
