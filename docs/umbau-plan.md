## Regeln für alle Phasen

1. **Lean-Prinzipien gelten weiter**: keine neuen Frameworks/Libraries ohne Notwendigkeit, Theme-Token-System unangetastet ([theme-token-system], [reject-webflow-frameworks]). Die Hausregel lautet neu: „kein React im **Prod-Output**" (nicht mehr „kein React im Projekt").
2. **Prod bleibt heilig**: Nach jedem Schritt muss `npm run build` (Prod-Config) ein rein statisches `dist/` ohne React-Runtime-Chunks und ohne Token erzeugen. Das ist das oberste Abnahmekriterium.
3. **Secrets**: Tokens niemals in Code, Config oder Git — nur `.env` (gitignored) lokal bzw. `wrangler secret put` remote. Skripte, die Secrets anfassen, geben aus WAS sie tun, aber nie die Werte.
4. Versionen der Sanity-Pakete nach dem Upgrade **exakt pinnen** (kein `^`), solange ggf. Patches getragen werden.
5. Am Ende jeder Phase: committen, kurzes Ergebnis-Protokoll unten in diesem Dokument ergänzen (Abschnitt „Protokoll").

## Phase A — Live-Core (der Brocken)

Ziel: Tippen im Studio erscheint in Millisekunden im Presentation-Preview, ohne Reload. Prod-Output unverändert statisch.

1. `@astrojs/react` + react 19 in der App; `sectionText.astro` + Section-Renderer → `.tsx`; `SectionsIsland`, die das `sections[]`-Array rendert.
2. Prod-Build: Island **ohne** `client:`-Directive (statisch gerendert, null React im Output). Preview-Build: `client:only`, geschaltet über die vorhandenen Preview-Gates (`PREVIEW_DEPLOY` / `#preview-extras`-Seam).
3. Initialdaten via `loadQuery`-Muster vom Server; im Presentation-iframe: **`usePresentationQuery` aus `@sanity/visual-editing/react`** (nicht `@sanity/react-loader` — Begründung Abschnitt 9 im Architektur-Doc). `useOptimistic` + `useDocuments().patch` für Insert/Duplicate/Move.
4. Click-to-edit: **explizite `data-sanity`-Attribute via `createDataAttribute`**, Stega aus (`stega: false`) — Remarkable-Muster, keine unsichtbaren Zeichen in Strings. Falls das im Astro-Kontext unerwartet hakt, ist Stega-an der dokumentierte Fallback.
5. Achtung (Remarkable-Learning): `<VisualEditing>`/Bridge **nicht code-splitten/lazy laden** — doppelte Modul-Globals killen die Optimistic-Updates still.
6. CORS/Preview-Handshake (`preview-url-secret`) bleiben wie vorhanden.

**Abnahme Phase A** (alles lokal: Studio `localhost:3333`, Preview `npm run dev:preview` auf `localhost:4321`):
- [ ] Tippen im Studio → Preview aktualisiert pro Tastenanschlag, ohne Full-Reload (sichtbar < ~100 ms)
- [ ] Drag-Reorder + Duplizieren/Löschen auf dem Canvas, sofortiges Feedback (optimistic)
- [ ] Klick auf Section-Inhalt im Preview öffnet das richtige Feld im Studio
- [ ] `npm run build`: `dist/` enthält keine React-Chunks, kein Token; Seiten byte-vergleichbar lean wie vorher
- [ ] i18n (DE/EN), Theme-Tokens, GSAP/Lenis funktionieren unverändert in Prod UND Preview
- [ ] Julian macht den Feel-Test selbst im Browser (Latenz ist subjektiv)

## Phase C — Cloudflare-Topologie

Architektur-Doc Abschnitt 5. Prod von Pages auf **Assets-only-Worker** (kein Adapter, kein `main`), Preview-Worker wie gehabt (SSR, `@astrojs/cloudflare`). Beide via Workers Builds am selben Repo, Sanity-Webhook → Deploy Hook. Nur fürs **Template** einrichten + dokumentieren — die drei Bestandsprojekte (frinighof-ws, vita-ws, ws-hasenegg: alle Prod=Pages + Preview-Worker) bleiben unangetastet.

**Abnahme:** Publish im Test-Studio → Prod-Worker deployt automatisch; Preview zeigt Drafts sofort; Prod-URL liefert statisch, ohne Token, anonyme Besucher der Preview-URL sehen nur Published.

## Phase D — One-Command-Provisioning

Erweitert `scripts/init.mjs` gemäß Architektur-Doc Abschnitt 6 (Sanity-Projekt, Dataset, CORS, Token, Webhook, Seed, Studio-Deploy, 2 Workers, Domains, Secrets). **Sicherheits-Leitplanken, von Julian vorgegeben:** lieber 2–3 manuelle Confirm-Schritte mehr als unsichere Automatik — d. h.: destruktive/kostenpflichtige Schritte (Domain-Zonen, Deploys auf Kundendomains) mit explizitem Confirm-Prompt; Dry-Run-Modus; Tokens nur via `wrangler secret put` / `.env`, nie geloggt, nie committed; idempotent (Re-Run darf nichts doppelt anlegen).

**Business-Entscheidungen (von Julian am 2026-07-09 entschieden, gelten als Default im Script):**
- **Org-Struktur: EINE Agentur-Org (upgreight), ein Sanity-Projekt pro Kunde.** Keine Org pro Kunde — Projekt-Transfer in eine Kunden-Org bleibt später jederzeit möglich, falls ein Kunde mal abwandert.
- **Plan: Free als Default.** Julian + Kunde sind beide Administrator (Free hat nur Admin/Viewer). Bewusst akzeptiertes Restrisiko: Ein Projekt-Admin kann in manage.sanity.io Mitglieder/Tokens/Datasets verwalten (Layout/Code kann er über Sanity nie ändern — das liegt im Repo). Bei euren kleinen, vertrauten Kunden okay (Frinighof bestätigt); Growth ($15/Seat, echte Editor-Rolle) nur als Upsell für heikle Kunden.
- Als Absicherung gegen versehentliches Löschen: Das Script dokumentiert/hinterlegt ein Backup-Kommando (`sanity dataset export`) pro Projekt, und der init-Lauf legt direkt nach dem Seed einen ersten Export ab.

**Abnahme:** Kompletter Durchlauf gegen ein frisches Wegwerf-Projekt: ein Befehl → funktionierende Studio-URL + Preview-URL + Prod-URL, danach rückstandsfrei löschbar. Zusätzlich zur normalen `/code-review`: **`/security-review` über das Provisioning-Script laufen lassen** (Token-/Secret-Handling), bevor es als fertig gilt.
